## 0. 들어가며

라이다든 카메라든, 센서는 절대 같은 값을 두 번 주지 않습니다. 똑같은 위치에 로봇을 세워놓고 거리를 100번 측정해도 결과는 매번 미세하게 다릅니다. 초보 엔지니어는 이걸 "노이즈니까 필터로 없애면 되겠지"라고 생각하지만, 실무에서는 노이즈를 없애는 게 아니라 **노이즈를 포함해서 확률적으로 다루는 법**을 배우는 게 먼저입니다.

특히 순찰 로봇처럼 "이상이 있는가/없는가"를 판단해서 경보를 울리는 시스템에서는, 센서 하나의 원시값을 그대로 믿는 순간 오탐(false alarm)과 미탐(missed detection) 사이에서 계속 사고가 납니다. 베이즈 통계는 이 문제를 다루는 가장 기본적이고 강력한 도구입니다.

## 1. 빈도주의 vs 베이즈 — 두 가지 사고방식

같은 질문 "이 센서의 오차 평균은 얼마인가?"에 대해 전혀 다르게 접근합니다.

| 구분 | 빈도주의 (Frequentist) | 베이즈 통계 (Bayesian) |
| --- | --- | --- |
| 알 수 없는 값을 보는 관점 | 고정된 하나의 참값이 존재한다고 가정 | 그 자체를 확률분포로 표현 |
| 새 데이터가 들어오면 | 다시 추정치를 계산 | 기존 분포를 갱신(update) |
| 결과물 | 점 추정값 하나 (예: 오차평균 = 0.03m) | 분포 전체 (예: 오차평균은 0.02~0.05m 사이일 가능성이 높다) |
| 로봇공학에서의 위치 | 센서 스펙 시트, 캘리브레이션 실험 | SLAM, 칼만 필터, 파티클 필터, 센서 퓨전 |

**현장 감각으로 비유하면:**
- 빈도주의는 "이 센서는 원래 ±3cm 오차를 갖는다"라고 스펙북에 적힌 숫자를 믿고 쓰는 것.
- 베이즈는 "지금까지 관측한 값들을 보니 이 벽까지 거리는 2.3m일 가능성이 가장 높고, 2.1~2.5m 사이일 확률이 90%다"라고 매 순간 갱신하는 것.

> SLAM, 칼만 필터, 파티클 필터, 오브젝트 트래킹(Kalman/EKF/UKF 기반)은 전부 베이즈 갱신의 응용입니다. "왜 로봇 알고리즘 이름에 확률·필터 이런 단어가 많이 들어가지?"라는 의문이 있었다면, 그 답이 여기 있습니다.

베이즈적 사고의 핵심 파이프라인은 이렇게 흘러갑니다.

```
기존 지식 (사전확률)
      ↓
새로운 센서 데이터 관측
      ↓
가능도 계산 (그 데이터가 각 가설에서 나올 확률)
      ↓
사후확률 = 갱신된 지식
```

이 사후확률이 다음 순간에는 다시 사전확률이 되어, 새 데이터가 들어올 때마다 계속 갱신됩니다. 이게 뒤에서 다룰 "순차 베이즈 갱신"입니다.

## 2. 확률의 기본 용어 — 현장에서 자주 쓰는 것 위주로

| 용어 | 정의 | 로봇공학 예시 |
| --- | --- | --- |
| 표본공간 | 발생 가능한 모든 결과의 집합 | 라이다 한 포인트가 가질 수 있는 모든 거리값 |
| 사건 | 관심 있는 결과의 부분집합 | "거리가 2m 이상" |
| 확률변수 | 무작위 결과를 숫자로 매핑한 변수 | 특정 순간의 거리 측정값 X |
| 모집단 | 분석 대상 전체 | 로봇이 하루 종일 수집하는 모든 센서값 |
| 표본 | 모집단에서 뽑은 일부 | 캘리브레이션 실험 중 수집한 1000개 측정값 |

**정확도 vs 정밀도**는 특히 자주 헷갈리는 개념인데, 현장에서는 이렇게 구분합니다.

- **정확도(Accuracy)**: 참값에 가까운가 → 캘리브레이션이 잘 됐는지의 문제
- **정밀도(Precision)**: 반복 측정값끼리 서로 가까운가 → 센서 자체의 노이즈 수준 문제

과녁 비유가 제일 직관적입니다. 화살이 과녁 중앙 근처에 몰려 있으면 정확도+정밀도 둘 다 좋은 것이고, 한쪽 구석에 촘촘히 몰려 있으면 정밀도는 좋지만 정확도가 나쁜 것 — 이 경우는 캘리브레이션(바이어스 보정)으로 고칠 수 있는 문제라 오히려 다행입니다. 화살이 사방에 흩어져 있으면 정밀도 자체가 나쁜 것이라 센서를 바꾸거나 필터링 전략을 다시 짜야 합니다.

**평균 / 분산 / 표준편차**도 매일 쓰게 되니 손에 익혀두는 게 좋습니다.

```python
import numpy as np

readings = np.array([2.31, 2.29, 2.34, 2.28, 2.30])  # 예: 거리 센서 5회 측정
mean = np.mean(readings)      # 중심값
var  = np.var(readings)       # 퍼짐 정도 (단위가 원본의 제곱)
std  = np.std(readings)       # 퍼짐 정도 (원본과 같은 단위 → 이게 실무에서 더 자주 쓰임)
```

> 노이즈 모델링할 때 report에는 분산(var)이 아니라 표준편차(std)를 쓰는 게 관례입니다. "센서 오차 분산이 0.0004"보다 "센서 오차 표준편차가 2cm"가 훨씬 직관적으로 와닿기 때문입니다.

## 3. 조건부 확률과 베이즈 정리

### 3.1 조건부 확률

"사건 B가 이미 일어났다는 조건 하에서, 사건 A가 일어날 확률"을 P(A|B)로 씁니다. 중요한 건 **P(A|B) ≠ P(B|A)** 라는 점입니다. 이걸 헷갈려서 생기는 실수가 로봇 안전 시스템에서 특히 치명적입니다.

- P(경보 | 실제 고장) — "고장이 났을 때 경보가 울릴 확률" (= 센서/모델의 감도)
- P(실제 고장 | 경보) — "경보가 울렸을 때 진짜 고장일 확률" (= 현장에서 그 경보를 얼마나 믿을 수 있는지)

이 둘은 완전히 다른 숫자입니다. 센서 스펙북에는 보통 첫 번째만 적혀 있는데, 현장 담당자가 실제로 궁금한 건 두 번째입니다. 이 간극을 메꾸는 게 베이즈 정리입니다.

### 3.2 베이즈 정리 유도

```
정의:
P(A|B) = P(A ∩ B) / P(B)
P(B|A) = P(A ∩ B) / P(A)

→ 양쪽의 P(A ∩ B)가 같으므로:
P(A|B) * P(B) = P(B|A) * P(A)

→ 정리하면 베이즈 정리:
P(A|B) = P(B|A) * P(A) / P(B)
```

| 기호 | 베이즈 용어 | 의미 |
| --- | --- | --- |
| P(A) | 사전확률 (prior) | 데이터를 보기 전 알고 있던 확률 |
| P(B\|A) | 가능도 (likelihood) | 그 상태(A)에서 이 데이터(B)가 관측될 확률 |
| P(B) | 증거확률 (evidence) | 데이터(B) 자체가 관측될 전체 확률 |
| P(A\|B) | 사후확률 (posterior) | 데이터를 본 후 갱신된 확률 |

### 3.3 로봇 순찰 시나리오로 계산해보기

화재 구역 감지 센서가 있다고 합시다.

- 사전확률: 특정 구역에서 실제로 화재가 날 확률 P(화재) = 0.001 (매우 드묾)
- 가능도: 화재가 났을 때 센서가 감지할 확률 P(경보|화재) = 0.98 (감도 98%)
- 오탐률: 화재가 아닌데 오경보가 뜰 확률 P(경보|정상) = 0.02

이때 "경보가 울렸을 때 진짜 화재일 확률" P(화재|경보)는 직관과 다르게 매우 낮게 나옵니다.

```python
p_fire = 0.001
p_alarm_given_fire = 0.98
p_alarm_given_normal = 0.02

p_evidence = p_alarm_given_fire * p_fire + p_alarm_given_normal * (1 - p_fire)
p_fire_given_alarm = (p_alarm_given_fire * p_fire) / p_evidence

print(round(p_fire_given_alarm, 4))  # 약 0.047 → 4.7%
```

경보가 울려도 실제 화재일 확률은 **4.7%**밖에 안 됩니다. 감도가 98%나 되는 좋은 센서인데도 말이죠. 이유는 화재 자체가 워낙 드문 사건(base rate)이라, 오탐이 조금만 섞여도 절대 개수로는 오탐이 진짜 화재보다 훨씬 많아지기 때문입니다.

> 이게 바로 "base rate fallacy"입니다. 경보 시스템을 설계할 때 "감도 98%면 훌륭하지"라고 끝내면 안 되고, 반드시 사전확률(사건의 실제 발생 빈도)까지 곱해서 "경보 하나가 실제로 얼마나 믿을만한가"를 확인해야 합니다. 다단계 경보(1차 의심 → 2차 확인 → 3차 최종 알람)를 두는 이유가 바로 이 낮은 사후확률을 끌어올리기 위함입니다.

## 4. 혼동행렬 — 경보 시스템의 성능표

앞서 계산한 네 가지 확률을 표 하나로 정리한 게 혼동행렬(Confusion Matrix)입니다.

| 실제 상태 \ 판정 | 경보 발생 | 경보 없음 |
| --- | --- | --- |
| **실제 이상** | TP (True Positive) | FN (False Negative) |
| **실제 정상** | FP (False Positive) | TN (True Negative) |

여기서 4개의 핵심 지표가 나옵니다.

| 지표 | 공식 | 확률 표현 | 의미 |
| --- | --- | --- | --- |
| 민감도 (Sensitivity/Recall) | TP / (TP+FN) | P(경보\|이상) | 진짜 이상을 얼마나 잘 잡아내는가 |
| 특이도 (Specificity) | TN / (TN+FP) | P(경보없음\|정상) | 정상 상황에서 오경보를 얼마나 안 내는가 |
| 정밀도 (Precision, PPV) | TP / (TP+FP) | P(이상\|경보) | 경보가 울렸을 때 실제로 믿을만한가 |
| 음성 예측도 (NPV) | TN / (TN+FN) | P(정상\|경보없음) | 경보가 없을 때 정말 안전하다고 믿어도 되는가 |

**두 종류의 실패는 로봇 안전 시스템에서 비용이 완전히 다릅니다.**

- **FN (미탐)**: 실제로 사람이 쓰러졌는데 경보가 안 울림 → 인명 사고로 직결, 가장 치명적
- **FP (오탐)**: 아무 일도 없는데 경보가 울림 → 현장 신뢰도 하락, 알람 피로(alarm fatigue)로 나중엔 진짜 경보도 무시하게 됨

> PPE 미착용 감지나 쓰러짐 감지 같은 안전 관련 모델은 threshold를 튜닝할 때 정밀도보다 민감도(Recall)를 우선하는 경우가 많습니다. 대신 오탐이 늘어나는 부작용은 2단계 검증(예: 연속 3프레임 이상 감지 시에만 최종 알람)으로 보완합니다. 목표 스펙을 "정밀도 95% 이상"으로만 잡지 말고, 반드시 민감도/특이도를 같이 명시하세요.

## 5. 순차 베이즈 갱신 — 센서 데이터를 시간에 따라 쌓기

로봇은 한 번만 측정하고 끝나지 않습니다. 매 프레임, 매 스캔마다 계속 관측이 들어오고, 그때마다 "믿음(belief)"을 갱신해야 합니다. 이게 순차 베이즈 갱신(Sequential Bayesian Update)입니다.

```
사전확률
   → 1번째 관측 → 사후확률 1
사후확률 1 (이게 다음 단계의 사전확률이 됨)
   → 2번째 관측 → 사후확률 2
사후확률 2
   → 3번째 관측 → 사후확률 3
   ...
```

앞의 화재 경보 예시를 연속으로 확장해봅시다. 경보가 연달아 3번 울렸다면 화재일 확률은 어떻게 변할까요?

```python
def bayes_update(prior, p_alarm_given_fire, p_alarm_given_normal):
    evidence = p_alarm_given_fire * prior + p_alarm_given_normal * (1 - prior)
    return (p_alarm_given_fire * prior) / evidence

belief = 0.001  # 최초 사전확률
for i in range(3):
    belief = bayes_update(belief, 0.98, 0.02)
    print(f"{i+1}번째 경보 후 화재 확률: {belief:.4f}")

# 1번째 경보 후: 0.0470
# 2번째 경보 후: 0.7069
# 3번째 경보 후: 0.9915
```

경보 한 번으로는 4.7%였던 확신이, 연속 3번 울리자 99% 이상으로 뛰어오릅니다. **이게 다단계/연속 알람 로직의 수학적 근거**입니다.

### 관측 독립성이라는 함정

위 계산은 매 경보가 서로 **조건부 독립**이라고 가정합니다. 하지만 실제 센서 데이터는 시간적으로 연관(correlated)되어 있는 경우가 많습니다. 예를 들어 카메라 프레임이 30fps로 들어온다면, 1초 사이의 30개 프레임은 거의 같은 정보를 담고 있어서 "30번 독립 관측"이 아니라 사실상 "1~2번 관측"에 가깝습니다.

> 이 가정을 무시하고 프레임마다 순진하게 베이즈 갱신을 돌리면, 실제로는 근거가 약한데도 확신도가 비정상적으로 빠르게 100%에 수렴하는 "overconfidence" 버그가 생깁니다. 실무에서는 (1) 다운샘플링해서 진짜 독립에 가까운 시점만 사용하거나, (2) 시간적 상관을 모델에 반영하는 HMM/칼만필터 계열로 넘어가거나, (3) 최소한 연속 감지에 "쿨다운 시간"을 둬서 이 문제를 완화합니다.

## 6. 확률분포 — 이산 vs 연속

### 6.1 이산확률분포

결과를 하나씩 셀 수 있는 경우입니다.

**베르누이분포** — 결과가 두 가지뿐인 한 번의 시행 (감지/미감지, 고장/정상)

- 평균: E[X] = p
- 분산: Var(X) = p(1-p)

**이항분포** — 같은 베르누이 시행을 n번 반복했을 때의 성공 횟수

- 평균: E[X] = np
- 분산: Var(X) = np(1-p)
- 조건: 시행 횟수 n 고정, 결과는 성공/실패 둘 중 하나, 성공확률 p 일정, 각 시행 독립

로봇 예시: 순찰 로봇이 하루 100번 특정 구역을 지나가면서 매번 2% 확률로 오탐을 낸다면, 하루 오탐 횟수는 이항분포 B(100, 0.02)를 따릅니다.

```python
from scipy.stats import binom

n, p = 100, 0.02
print(binom.pmf(3, n, p))   # 오탐이 정확히 3번 일어날 확률
print(binom.cdf(3, n, p))   # 오탐이 3번 "이하"일 확률
```

### 6.2 연속확률분포와 정규분포

연속분포는 특정 한 점의 확률이 아니라 **구간의 확률**을 다룹니다. `P(X = 3.0) = 0`이지만 `P(2.9 ≤ X ≤ 3.1)`은 의미 있는 값을 가집니다. 센서 노이즈는 거의 대부분 정규분포로 모델링합니다.

```
평균 ± 1표준편차: 약 68%
평균 ± 2표준편차: 약 95%
평균 ± 3표준편차: 약 99.7%
```

```python
from scipy.stats import norm

mean, std = 2.30, 0.02  # 예: 거리센서 평균 2.30m, 표준편차 2cm
p = norm.cdf(2.34, mean, std) - norm.cdf(2.26, mean, std)  # 2.26~2.34m 구간 확률
```

### 6.3 이산 vs 연속 요약

| 구분 | 이산분포 | 연속분포 |
| --- | --- | --- |
| 대표 분포 | 베르누이, 이항 | 정규분포 |
| 확률 함수 | PMF (확률질량함수) | PDF (확률밀도함수) |
| 한 점의 확률 | 계산 가능 | 항상 0 |
| 구간 확률 계산 | 값들의 합 | 곡선 아래 면적(적분) |
| SciPy 함수 | `binom.pmf()` | `norm.pdf()` |
| 누적확률 함수 | `binom.cdf()` | `norm.cdf()` |
| 로봇 적용 예시 | 하루 오탐 횟수, n회 중 감지 성공 횟수 | 거리/속도/각도 센서 노이즈 |

### 6.4 Z-점수 — 이상치를 빠르게 잡는 법

```
Z = (측정값 - 평균) / 표준편차
```

측정값이 평균에서 표준편차 몇 개만큼 떨어져 있는지를 나타내는 표준화된 점수입니다. 보통 `|Z| > 3`이면 이상치로 의심합니다.

> Z-score의 한계는 평균과 표준편차 자체가 이상치의 영향을 받는다는 점입니다. 센서에 튀는 값이 섞여 있으면 평균과 표준편차가 같이 튀어서, 정작 그 이상치를 놓치는 역설이 생깁니다. 이상치가 많이 섞일 것으로 예상되는 데이터(예: 초기 캘리브레이션 전 raw 데이터)에는 중앙값(median)과 MAD(Median Absolute Deviation) 기반의 로버스트 Z-score를 대신 쓰는 걸 권장합니다.
