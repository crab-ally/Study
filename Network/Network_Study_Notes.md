# Part 1. 기초 — 내 컴퓨터가 인터넷에 연결되는 원리

## 1. IP 주소 (IP Address) — 네트워크 세상의 행정구역 주소

> 💡 이사를 가면 우편 주소가 바뀌듯, 네트워크 환경(와이파이, 회사, 카페)이 바뀌면 IP도 바뀝니다.

IP 주소는 인터넷이나 네트워크에서 장치들을 서로 구분하기 위해 부여하는 **논리적인 주소**입니다.

### 1-1. 주소 체계의 형태

| 구분 | 형태 | 특징 |
|---|---|---|
| **IPv4** | `192.168.0.1` | 32비트 숫자 주소. 가장 널리 쓰이지만 **주소 고갈** 문제 발생 |
| **IPv6** | `2001:0db8:85a3:0000:0000:8a2e:0370:7334` | 128비트 차세대 주소 체계 |

> 🔧 국내 대부분의 사내망은 여전히 IPv4 기반입니다. IPv6는 알아만 두시고, 실무에서는 IPv4 계산(서브넷 마스크 등)에 훨씬 더 많이 부딪히게 될 겁니다. 서브넷 마스크는 8장에서 자세히 다룹니다.

### 1-2. 공인 IP vs 사설 IP

* **공인 IP (Public IP):** ISP가 할당하는, 전 세계에서 유일한 "진짜 주소". 외부에서 접속하려면 반드시 필요
* **사설 IP (Private IP):** 내부망(공유기 안쪽)에서만 쓰는 주소 (`192.168.x.x`, `10.x.x.x`). 외부에서 직접 찾아올 수 없음

> 🔧 "내 컴퓨터 IP가 192.168.0.5인데 왜 외부에서 접속이 안 되나요?" — 정말 많이 받는 질문입니다. **사설 IP**라서 인터넷 바다에서는 아예 보이지 않기 때문입니다. 이걸 해결해주는 기술이 7장에서 다룰 **NAT**입니다.

### 1-3. 내 주소 직접 확인해보기

* **공인 IP:** 포털 사이트에 "내 IP 주소" 검색
* **사설 IP:** Windows는 `ipconfig`, Mac/Linux는 `ifconfig` 또는 `ip a`

## 2. MAC 주소 — 기기의 주민등록번호

> 💡 IP가 "행정구역 주소"라면, MAC은 "주민등록번호"입니다. 전 세계에 단 하나뿐이고 변경 불가능합니다.

| 구분 | MAC 주소 (물리 주소) | IP 주소 (논리 주소) |
|---|---|---|
| 목적 | 하드웨어 식별 | 네트워크 위치 식별 |
| 변경 가능성 | 변경 불가 | 상황에 따라 변경 가능 |
| 할당 방식 | 제조사가 생산 시 지정 | 관리자 지정 또는 DHCP 자동 할당 |
| OSI 계층 | 2계층 (데이터 링크 계층) | 3계층 (네트워크 계층) |

> **핵심:** 먼 곳(인터넷)을 찾아갈 때는 IP, 같은 방(LAN) 안에서 옆 기기로 전달할 때는 MAC.

> 🔧 공유기의 "MAC 주소 필터링"은 등록된 기기만 와이파이 접속을 허용하는 보안 기능입니다. IP는 바뀌어도 MAC은 고정이기 때문에 가능합니다.

## 3. 랜선 (LAN 케이블) — 데이터가 실제로 달리는 물리적 도로

### 3-1. 규격별 종류

| 규격 | 속도 | 최대 거리 | 주 사용처 |
|---|---|---|---|
| **Cat.5e** | 1Gbps | 100m | 일반 가정, 일반 사무실 |
| **Cat.6** | 1~10Gbps | 100m (10Gbps는 55m 이내) | 신축 건물·사무실 표준 |
| **Cat.6a** | 10Gbps | 100m | 기업용 백본, 데이터 센터 |
| **Cat.7 / Cat.8** | 10~40Gbps | - | 노이즈 심한 공장, 특수 서버실 |

### 3-2. 차폐 방식

* **UTP:** 보호막 없음. 가정/사무실에 충분
* **FTP:** 알루미늄 호일로 전체 감싸 잡음 방지
* **STP:** 페어별 편조 쉴드. 공장·대형 서버실에 필수

> 🔧 인터넷 속도가 갑자기 느려졌다면 소프트웨어보다 먼저 케이블 상태부터 확인하세요.

## 4. 전체 흐름 정리 — 내 컴퓨터가 구글에 접속할 때

```
[내 PC] --(랜선/와이파이)--> [공유기] --(NAT)--> [인터넷] --> [구글 서버]
```

1. 내 컴퓨터가 구글의 **공인 IP**(`142.250.x.x`)로 가고 싶은 패킷을 만든다.
2. 공유기에게 전달하기 위해 공유기의 **MAC 주소**를 찾아 2계층으로 넘긴다.
3. 공유기는 **NAT**로 내 **사설 IP**를 자신의 **공인 IP**로 변환해 인터넷으로 던진다.
4. 여러 라우터를 거쳐 구글의 **공인 IP**에 도착한다.

> 📌 사실 우리는 브라우저에 `142.250.x.x`가 아니라 `google.com`을 입력합니다. 이 변환 과정이 Part 2의 **DNS**입니다.

## 5. TCP/IP 프로토콜 — 통신의 규칙과 약속

**프로토콜(Protocol)**이란 컴퓨터나 장치들이 서로 데이터를 주고받는 규칙과 약속입니다.

### 5-1. 대표적인 프로토콜

| 프로토콜 | 설명 |
|---|---|
| **HTTP** | 웹사이트를 볼 때 사용 (자세한 보안 이슈는 12장에서) |
| **TCP/IP** | 인터넷의 기본 프로토콜. IP는 주소를 찾고, TCP는 정확한 전달을 보장 |
| **FTP** | 파일을 주고받을 때 사용 |

### 5-2. TCP vs UDP — 전화 통화 vs 문자 메시지

> **TCP는 전화 통화**입니다. 상대방이 잘 받았는지 계속 확인하면서 대화합니다.  
> **UDP는 문자 메시지**입니다. 일단 보내고, 받았는지 신경 쓰지 않고 계속 보냅니다.

| 구분 | TCP | UDP |
|---|---|---|
| 연결 여부 | 연결형 | 비연결형 |
| 신뢰성 | 높음 (재전송) | 낮음 (손실 시 무시) |
| 순서 | 보장 | 보장 안 함 |
| 속도 | 느림 | 빠름 |
| 예시 | 웹사이트, 이메일, 파일 전송, 채팅 | 스트리밍, 온라인 게임 |

### 5-3. TCP의 누락 패킷 복구 원리

1. **감지** — 순서 번호(sequence number)로 빠진 패킷 파악
2. **ACK 기반 요청** — 마지막으로 연속 수신한 패킷까지만 ACK, 일정 시간 응답 없으면 손실 간주
3. **재전송** — 손실 패킷만 재전송 후 순서대로 재조합

> 🔧 화상회의는 UDP라서 살짝 깨져도 안 끊기고, 파일 다운로드는 TCP라서 한 바이트도 손실 없이 받습니다.

## 6. 포트(Port)와 쿠키(Cookie)

### 6-1. 포트 — 같은 주소, 다른 서비스 구분

`192.168.1.100:80` → IP(`192.168.1.100`)는 "아파트 동 주소", 포트(`80`)는 "몇 호"

> 🔧 방화벽의 "포트를 연다/닫는다"는 14장에서 자세히 다룹니다.

### 6-2. 쿠키 — 웹사이트가 나를 기억하는 방법

**역할:** 로그인 상태 유지, 개인화 설정, 장바구니 기억, 이용 통계 수집
**작동 순서:** 방문 → 서버가 쿠키 생성 요청 → 재방문 시 브라우저가 쿠키 전달 → 서버가 개인화된 콘텐츠 제공

> 🔧 "로그인이 자꾸 풀려요" 문의는 대부분 쿠키 차단이나 짧은 만료 시간이 원인입니다.

# Part 2. 심화 — 인터넷과 라우팅

## 7. DNS — 도메인을 IP로 바꾸는 통역사

> 💡 DNS는 전화번호부입니다. 사람 이름(도메인)만 알면 실제 번호(IP)는 찾아줍니다.

```
ychoi.kr (도메인) → DNS → 185.199.110.153 (IP)
```

**동작 순서:** 주소 입력 → DNS 서버에 IP 요청 → DNS가 IP 응답 → 브라우저가 해당 IP로 접속

**기타 역할:** 하위 도메인 설정, 메일 서버 위치 지정(MX 레코드), 서비스 위치 지정(SRV 레코드), 보안 설정(DNSSEC)

> 🔧 "사이트가 안 열려요" 장애의 상당수는 서버가 죽은 게 아니라 **DNS 응답 실패**입니다. `nslookup 도메인명`이나 `dig 도메인명`으로 먼저 확인하는 게 첫 점검 습관입니다.

## 8. NAT 심화 — 공인 IP를 아껴 쓰는 기술

```
| 내부 장치들 (사설 IP) | ⇄ NAT 장비(공유기) ⇄ | 인터넷 (공인 IP) |
```

**쓰는 이유:** ① 공인 IP 부족 해결 ② 보안성 강화(내부 장치 비노출, 방화벽 역할 겸함) ③ 내부망 관리 용이

| 종류 | 방식 | 주 사용처 |
|---|---|---|
| **Static NAT** | 내부 IP ↔ 공인 IP 1:1 고정 매핑 | 외부에 항상 노출돼야 하는 서버 |
| **Dynamic NAT** | 다수 내부 IP ↔ 공인 IP 풀(pool) 매핑 | 공인 IP를 여러 개 보유한 기업망 |
| **PAT** | 공인 IP 하나를 포트 번호로 다수 내부 IP가 공유 | 가정용 공유기 (가장 흔함) |

> 🔧 집 공유기에 여러 기기가 동시에 인터넷을 쓸 수 있는 건 **PAT** 덕분입니다. 각 기기의 요청을 포트 번호로 구분해서 관리합니다.

## 9. 서브넷 마스크 — 네트워크를 쪼개는 기준

```
IP 주소 = [네트워크 영역 | 호스트 영역]
```

`192.168.10.20` + 서브넷 마스크 `255.255.255.0` → 네트워크: `192.168.10`, 호스트: `20`

**나누는 이유:** ① 네트워크 분할 ② IP 효율적 사용 ③ 트래픽 제어 및 보안 향상

| 서브넷 마스크 | CIDR | 설명 |
|---|---|---|
| 255.0.0.0 | /8 | 네트워크 8비트 |
| 255.255.0.0 | /16 | 네트워크 16비트 |
| 255.255.255.0 | /24 | 네트워크 24비트 (**가장 흔함**) |
| 255.255.255.128 | /25 | 네트워크 25비트 |

> 🔧 "개발팀 192.168.1.0/24, 인사팀 192.168.2.0/24"처럼 부서별로 대역을 나누면 트러블슈팅과 보안 정책 적용이 훨씬 쉬워집니다.

## 10. QoS — 트래픽에도 우선순위가 있다

**주요 기능:** 대역폭 관리, 우선순위 관리, 지연/지터 관리, 패킷 손실 관리

**주의사항:** 모든 장비가 지원하는 건 아니며, 잘못 설정하면 오히려 성능이 떨어질 수 있음

> 🔧 화상회의는 끊기는데 가족이 대용량 파일을 받고 있다면, 공유기 QoS에서 화상회의 트래픽에 우선순위를 부여해보세요.

---

## 11. 망중립성 — 인터넷을 다루는 원칙

통신사가 모든 트래픽을 **차별 없이** 처리해야 한다는 원칙입니다.

**핵심:** 차별금지 · 동등 접근 · 공정성

> 📌 QoS는 "내 네트워크 안"의 우선순위 기술, 망중립성은 "통신사가 인터넷 전체"를 다룰 때 지켜야 할 원칙입니다.

# Part 3. 네트워크 보안 — 안전하게 연결한다는 것

지금까지는 "연결이 되는 원리"를 배웠습니다. 이제부터는 "그 연결이 왜 위험할 수 있고, 어떻게 지켜야 하는지"를 다룹니다. 현업에서 장애 대응보다 더 긴장되는 순간이 바로 보안 사고 대응입니다.

## 12. IP 주소 노출의 위험

IP 주소 하나가 노출되는 것만으로도 다음과 같은 위험이 따라옵니다.

* **개인 위치 노출** — 대략적인 거주 지역이나 도시가 유추될 수 있음
* **해킹·사이버 공격 위험 증가** — 공격자가 IP를 이용해 DDoS 공격이나 브루트포스(무차별 암호 대입)를 시도하기 쉬워짐
* **사생활 침해** — 인터넷 활동 일부가 추적당할 가능성
* **악의적 공격·괴롭힘의 표적** — 특정인을 노린 지속적인 공격 시도

> 🔧 온라인 게임이나 화상 통화 서비스에서 상대방의 IP가 노출되는 구조라면, 그 서비스는 반드시 IP를 가리는 중계 서버(릴레이)를 거치도록 설계해야 합니다.

## 13. HTTP와 HTTPS — 데이터를 그냥 보낼까, 잠가서 보낼까

| 항목 | HTTP | HTTPS |
|---|---|---|
| 프로토콜 | HyperText Transfer Protocol | HTTP + Secure (TLS/SSL) |
| 기본 포트 | 80 | 443 |
| 암호화 | ❌ 없음 (평문 전송) | ✅ 암호화 전송 |
| 보안성 | 낮음 (도청·변조 가능) | 높음 (도청·변조 방지) |
| 인증서 | 필요 없음 | SSL/TLS 인증서 필요 |
| URL | `http://` | `https://` |

> 💡 SSL/TLS는 공개키 암호화와 대칭키 암호화를 조합해 데이터를 안전하게 전송합니다.

### 13-1. HTTPS 연결 과정

```
1. 클라이언트(브라우저)가 서버에 접속 요청
2. 서버가 SSL/TLS 인증서 제공
3. 클라이언트가 인증서를 검증하고 암호화된 터널 생성
4. 이후 모든 데이터는 이 터널을 통해 안전하게 전송
```

### 13-2. 왜 중요한가 — 로그인 예시

* HTTP로 전송 시 → ID와 비밀번호가 **암호화되지 않은 채** 그대로 전달됨 (위험!)
* HTTPS로 전송 시 → ID와 비밀번호가 **암호화**되어 전달됨 (안전)

> 🔧 주소창에 자물쇠 아이콘이 없고 `http://`로 시작하는 사이트에서는 절대 로그인 정보나 카드번호를 입력하지 마세요. 요즘은 거의 모든 브라우저가 HTTP 사이트에 경고를 띄웁니다.

## 14. Wi-Fi 암호화 — 무선 구간을 지키는 방법

| 암호화 방식 | 보안 수준 | 설명 |
|---|---|---|
| WEP | 🔴 매우 취약 | 가장 오래되고 취약함 (절대 사용 금지) |
| WPA | 🟡 보통 | WEP보다 나아졌지만 여전히 위험 |
| WPA2 | 🟢 우수 | 현재 가장 널리 쓰이며 충분히 안전 |
| WPA3 | ✅ 매우 우수 | 최신 방식, WPA2보다 강화됨 (권장) |

> 🔧 WPA2를 쓸 때는 반드시 **AES 암호화 모드**를 선택하세요. 구형 TKIP 모드가 남아있는 공유기가 의외로 많습니다.

## 15. 방화벽 (Firewall) — 네트워크의 출입문

외부에서 들어오는 트래픽과 내부에서 나가는 트래픽을 감시하고, 규칙에 따라 차단하거나 허용하는 장치 또는 소프트웨어입니다.

### 15-1. 주요 역할

* **트래픽 검사** — 오가는 모든 데이터를 검사·필터링
* **접근 제어** — 지정된 규칙(허용/차단)에 따라 통제
* **악의적 접근 방지** — 해커나 악성코드로부터 내부망 보호

### 15-2. 작동 방식

1. **패킷 필터링** — IP, 포트, 프로토콜 기준으로 차단/허용
   ```
   예: 외부에서 들어오는 포트 80(HTTP)은 허용
   예: 외부에서 들어오는 포트 23(Telnet)은 차단
   ```
2. **상태 기반 검사 (Stateful Inspection)** — 연결 상태를 기억해 정상/비정상 요청 구분
3. **프록시 방화벽** — 클라이언트와 서버 사이에서 중계 역할
   ```
   클라이언트 ↔ [프록시 방화벽] ↔ 인터넷
   ```

### 15-3. 종류

* **하드웨어 방화벽** — 독립된 물리 장치, 네트워크 입구에서 관리 (예: 기업용 방화벽 장비)
* **소프트웨어 방화벽** — OS에 설치해 개별 컴퓨터를 보호 (예: Windows 방화벽, 리눅스 iptables)

### 15-4. 실제 사용 예시

* 외부에서 SSH, FTP 접속을 막기 위해 특정 포트 차단
* 내부 직원만 중요 시스템에 접근하도록 IP 제한
* 웹 서비스는 포트 80, 443만 허용하고 나머지는 전부 차단

> 🔧 신규 서버를 세팅할 때 가장 먼저 하는 일이 "필요한 포트만 열고 나머지는 다 닫는 것"입니다. 기본값 그대로 모든 포트를 열어두는 건 현업에서 가장 흔한 사고 원인 중 하나입니다.

## 16. VPN (Virtual Private Network) — 인터넷 속 나만의 안전한 터널

인터넷상에 가상 터널을 만들어 데이터를 안전하고 비공개로 주고받게 해주는 기술입니다.

### 16-1. 핵심 원리

* **터널링** — 공개된 인터넷 위에 보안된 경로(터널)를 만들어, 외부에서는 그 안의 내용을 볼 수 없음
* **암호화** — 터널 안의 데이터를 암호화해 제3자가 볼 수 없게 함 (터널 양 끝에서만 복호화)

### 16-2. 작동 과정

```
1. 사용자 장치가 VPN 서버에 접속 요청
2. 사용자 장치 ↔ VPN 서버 사이에 암호화된 터널 생성
3. 사용자 데이터가 암호화되어 터널을 통해 VPN 서버로 이동
4. VPN 서버가 복호화 후 실제 목적지(웹사이트 등)로 전달
   → 이때 목적지는 사용자의 실제 IP가 아닌 VPN 서버의 IP만 확인 가능
5. 목적지의 응답이 VPN 서버로 전달
6. VPN 서버가 응답을 다시 암호화해 사용자에게 전송
7. 사용자 장치가 복호화하여 최종 데이터 수신
```

> VPN 서버를 경유하고, 암호화·복호화 작업이 추가되며, VPN 자체의 속도 제한이나 서버 혼잡 등의 영향으로 일반적인 인터넷 연결보다 통신 속도가 느려질 수 있습니다.

### 16-3. 사용하는 이유

* **보안 및 프라이버시 강화** — 암호화로 도청·해킹 방지, IP 은닉으로 개인정보 보호
* **지리적 제한 우회** — 다른 국가 IP로 제한된 콘텐츠에 접근
* **원격 업무 지원** — 집이나 외부에서 회사 내부망에 안전하게 접속

### 16-4. 활용 사례

해외에서 국내 서비스 이용, 공공 Wi-Fi에서의 보안 유지, 기업의 원격 근무 환경

> 🔧 카페나 공항 같은 공공 Wi-Fi는 암호화가 약하거나 아예 없는 경우가 많습니다. 회사 업무를 처리해야 한다면 반드시 VPN을 켜고 접속하는 습관을 들이세요.

## 17. DDoS (Distributed Denial of Service) — 분산 서비스 거부 공격

여러 대의 컴퓨터나 장치를 이용해 하나의 서버에 동시에 과도한 트래픽을 발생시켜, 정상 서비스를 마비시키는 공격입니다.

### 17-1. 특징

* 수백~수만 대의 기기로 대량 요청을 동시에 전송
* 서버의 CPU, 메모리 등 리소스를 소진시켜 정상 사용자를 막음
* 공격 출처가 분산되어 있어 방어가 어려움

### 17-2. 공격 원리

```
1. 공격자가 악성코드를 여러 컴퓨터/IoT 장치에 감염시켜 봇넷을 형성
2. 명령을 내려 특정 서버로 트래픽을 집중시킴
3. 대상 서버가 트래픽을 처리하지 못하고 과부하로 다운됨
```

### 17-3. 목적

특정 웹사이트를 다운시키는 업무방해, 쇼핑몰·게임 서버를 노린 경제적 피해 유발, 정치·사회적 메시지 전달

### 17-4. 방어 방법

* **트래픽 필터링과 방화벽 강화** — 비정상 트래픽 차단
* **DDoS 방어 전용 서비스 활용** — Cloudflare, AWS Shield, Akamai 등
* **서버 용량 및 네트워크 증설** — 순간적 트래픽 급증 대응
* **IP·포트 관리 강화** — 허용된 IP만 접근하도록 제한

### 17-5. DoS vs DDoS

| 유형 | 설명 | 공격 원리 |
|---|---|---|
| **DoS** | 단일 컴퓨터가 공격 | 단일 출처에서 서버 공격 |
| **DDoS** | 다수의 컴퓨터가 공격 | 여러 출처에서 동시 공격 |

> 🔧 DDoS는 예방보다 "탐지 후 얼마나 빨리 대응하느냐"가 관건입니다. 트래픽 패턴을 평소에 모니터링해두면, 이상 징후를 훨씬 빨리 잡아낼 수 있습니다.
